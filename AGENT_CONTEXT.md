# AGENT_CONTEXT.md — Handoff for Copilot Sessions

This file is the **persistent memory** for Copilot agent sessions on the AzureBackup repo. It is checked into source control so the agent can read it on a fresh chat (on **any machine, any OS-supported environment**) and immediately understand the project state, conventions, recent decisions, and open questions.

## How to use this file

**At the start of every fresh Copilot chat:** read this file before doing anything else. Treat the "Active workstreams" section as a continuation of the previous session.

### Pre-commit checklist (run this immediately before every `git commit`)

This section is the load-bearing rule of the file. **The agent has demonstrably skipped it before.** When that happens the file silently drifts and the next session inherits a lie. Treat it as non-optional.

Before running `git add .; git commit -m "..."`, walk through these questions in order. If the answer to any is "yes", the commit MUST also include an AGENT_CONTEXT.md edit (and frequently a `docs/USER_GUIDE.md` or `docs/SETUP.md` edit per the Documentation trust policy).

1. **Did the change introduce a new architectural fact?** New `public` API on a `Core` type, new cross-project constant, new env var read at startup, new build-time `DefineConstants`, new file-format magic number / version byte, new partial-class seam, new singleton, new background task, new on-disk file location, new lock or shared-state contract. → Add an entry to the "Architectural facts to load into working memory" list. Do not assume "it's just a refactor" — a public API / public constant / cross-boundary contract IS a fact.
2. **Did the change introduce or remove a hard-coded duplicate of a value used in more than one project?** → Add a fact recording the canonical location and a "do not introduce a new copy" warning. (See fact #9 for the template.)
3. **Did the change land a commit with a B-number?** → Add a row to the "Recent commit history" table. Add an entry to "Completed workstreams". If the commit consumed a B-number that was pre-allocated to a different planned task in "Active workstreams", renumber the planned task and add a one-line note explaining the renumbering.
4. **Did the change resolve, block, or pivot any workstream?** → Update the corresponding W-numbered section. If a workstream is now done, move it to "Completed workstreams".
5. **Did the change establish or change a user preference, coding convention, or workflow rule?** → Add or update the corresponding entry under "Hard rules".
6. **Did the change discover a non-obvious production behavior** (memory, concurrency, persistence, network, encryption)? → Add a fact.
7. **Did the change produce a benchmark number that affects a design decision?** → Update the relevant benchmark file's xmldoc table (the source of truth) AND add a one-line summary in the relevant W-numbered section. Do not paste the full numbers here.
8. **Did the change teach a "do not do this" lesson?** → Add a fact, with the word "NEVER" or "DO NOT" so a future session greps for it.
9. **Did this session touch AGENT_CONTEXT.md at all?** → Add a dated bullet to the "Maintenance log for THIS file" at the bottom summarizing what changed in the file. One bullet per session, not per edit.

If the answer to all nine is "no", commit without touching this file. That is the only acceptable reason to skip the update.

### Hidden failure mode that has happened before

The trap is reasoning "this commit only changes code, it doesn't change documentation, so AGENT_CONTEXT doesn't need an update." That logic is wrong because **AGENT_CONTEXT documents the code, not the documentation**. A code commit that adds a new public constant is exactly the trigger for question 1 above. Commit `b9d5744` (B26) hit this trap; the catch-up was commit `df0df53` and cost the user an extra prompt to ask "did you remember the file?". Do not repeat it.

### Implausibility-prompts-verification (added 2026-04-27 after B43)

When a user reports a measurement, log line, or behaviour whose magnitude looks *implausible* given the documented design (e.g. "throughput shows 6 Mbps on gigabit Azure", "memory used reads 41 GB on a 32 GB host", "auto-repair reset to zero after restart"), do NOT accept it at face value AND do NOT immediately blame the user's environment. Treat implausibility as a prompt to verify the LABEL against the FORMULA at every producer in the codebase before tuning anything. The B43 throughput rename was the textbook case: production logs showed values that looked like single-digit Mbps for a gigabit pipeline, but the on-disk number was actually MB/s -- a factor-of-8 label error that would have wasted a network-perf investigation. The check costs one `code_search` for the field name; the cost of skipping it is days of misdirected debugging.

### Edit policy (style)

- Keep entries dated (YYYY-MM-DD).
- Move completed workstreams to the "Completed workstreams" section near the bottom.
- Keep the file under ~600 lines so a fresh agent can read it in one tool call.
- **Never** record machine-specific values (free disk, CPU model, core count, terminal-tool timeouts, exact wall-clock times, OS-edition build numbers). Those mislead the next session if it is on a different PC. Record *relative* findings (deltas, percentages, comparisons) and *workload* identity (which benchmark / which workload name produced the number) instead.

**Anti-pattern:** do not duplicate code or full xmldoc into this file. Reference file paths, commit hashes, and benchmark file xmldoc tables instead.

## Project at a glance

- **Name:** AzureBackup — a cross-platform desktop application that backs up local files to Azure Blob Storage with client-side encryption and content-defined chunking for deduplication. Runs on Windows, macOS, and Linux.
- **Language/runtime:** C# on .NET 10 (`net10.0` TFM).
- **Solution:** `azurebackup.sln` at repo root.
- **Project layout:**
  - `src/AzureBackup.Core/` — all backup/restore/encryption/database logic. Cross-platform-clean C# library; no UI dependencies. **All non-trivial logic lives here.**
  - `src/AzureBackup/` — **Avalonia 11.3.12** desktop UI shell (`MainWindowViewModel` + views). Targets `net10.0` (no `-windows` suffix), so it builds and runs on Windows, macOS, and Linux. Verify with `Select-String Avalonia src/AzureBackup/AzureBackup.csproj` if in doubt.
  - `tests/AzureBackup.Tests/` — xUnit. As of B25-bench-2: 759 tests, all passing.
  - `benchmarks/AzureBackup.Benchmarks/` — BenchmarkDotNet 0.15.8. Release-only, never run in CI. See "Running tests and benchmarks" below.
- **Setup, build, publish, file locations:** `docs/SETUP.md` is current and authoritative as of the most recent commit that touched it. If you change build flags, target frameworks, runtime identifiers, the `portable.marker` mechanism, the `AppMode.DataDirectory` resolution, or the encryption/chunking parameters listed in its "Technical specifications" table, you MUST update `docs/SETUP.md` in the same commit.
- **End-user UX (views, settings, menus, dialogs):** `docs/USER_GUIDE.md` is current and authoritative as of the most recent commit that touched it. If you add or rename a view (`*.axaml` under `src/AzureBackup/Views/`), add or rename a button/toggle/setting, change a default value visible to the user, or change a user-visible workflow, you MUST update `docs/USER_GUIDE.md` in the same commit.
- **Database backend:** SQLCipher-encrypted SQLite (production default since commit `63103b1` = `C-5`). Legacy LiteDB code path still present for migration only.
- **Authentication:** Argon2id KDF (64 MB, 8 lanes, 3 iterations) for both database key derivation and the `EncryptionService` content key.
- **Encryption envelope:** AES-256-GCM with `[magic(4) | version(1) | nonce(12) | ciphertext(N) | tag(16) | crc32(4)]`. `EncryptionService.EncryptionOverhead = 37`.
- **Chunking:** content-defined (Rabin-style rolling hash, window=48, prime=31). Per-extension chunk-size config (`.mkv` uses 1-64 MB, `.txt` uses 16-128 KB, etc.). Files >500 MB use 16-128 MB chunks regardless of extension.

## Hard rules (USER PREFERENCES — never violate)

These come from `.github/copilot-instructions.md` and from explicit user instruction during session work. Reproduced here because they apply to every session.

### Documentation trust policy

**Treat every Markdown documentation file in this repo as stale until proven current** — with exceptions noted below.

**Verified-current docs (treat as authoritative, but maintain rigorously):**

- `docs/SETUP.md` — verified accurate against source on 2026-04-24. Covers Azure provisioning, build-from-source, single-file publish, portable mode (`portable.marker`), encryption envelope, per-extension chunking config, storage tiers, `AppMode.DataDirectory` paths, and a verified "Technical specifications" table.
- `docs/USER_GUIDE.md` — verified accurate against source on 2026-04-24. Covers every view (`SyncView`, `SettingsView`, `StorageHealthView`, `TierMigrationView`, `DataIntegrityView`, `LogsView`), the diagnostic-bundle export flow, the memory-limit slider, and all four storage tiers (Hot/Cool/Cold/Archive).
- `AGENT_CONTEXT.md` - verified accurate against source on 2026-04-27. Covers project layout, architectural facts, workstream state, and the conventions for commits, terminal commands, and tool usage that every future session MUST follow.

**Mandatory maintenance protocol for the verified-current docs:** every commit that touches one of the things they describe MUST update them in the same commit. Specifically:

- Adding/renaming a view file (`src/AzureBackup/Views/*.axaml`) → update `docs/USER_GUIDE.md` table of contents and the corresponding section.
- Adding/renaming a user-visible button, toggle, or setting → update `docs/USER_GUIDE.md`.
- Changing a default value the user can see (default storage tier, default container name, default exclusion patterns, default memory-limit state) → update `docs/USER_GUIDE.md`.
- Changing the encryption envelope, Argon2id parameters, chunk-size config, storage-tier set, or `AppMode.DataDirectory` resolution → update `docs/SETUP.md`.
- Changing build flags, target framework, runtime identifiers, the publish profile, or NuGet package versions called out in the technical-specifications table → update `docs/SETUP.md`.
- Adding or renaming a top-level menu, tab, or workflow step → update both docs as appropriate.
- **Renaming a serialized field, schema column, JSON key, log key, or any other on-disk / on-wire identifier** → update every doc that quotes the old name in the same commit. The grep target is the literal old token, not the prose around it; doc table cells that reference the old key are silent landmines until found.

**Label/formula consistency rule (added 2026-04-27 after B43).** Any persisted numeric field whose NAME implies a unit (e.g. `*_mbps`, `*_kib`, `*_us`, `*_per_sec`) MUST match the formula at every producer. The agent shipped `ThroughputMbps` for months while every producer divided by `1024 * 1024` (i.e. produced megabytes per second, MB/s) -- the label was wrong by a factor of 8 and silently corrupted every downstream throughput comparison. Do not trust the existing field name as evidence of the unit; trace the formula at the producer and the consumer separately and confirm they agree. When renaming the field, pin the JSON key explicitly via `[JsonPropertyName(...)]` so the configured naming policy cannot silently regress the key on a future rename.

When the doc and the code drift, the doc is wrong by definition; fix the doc, do not weaken the code to match. If a commit cannot reasonably update the doc in the same change (e.g. an emergency hotfix), record an observation in this file's maintenance log and open a follow-up.

For any other Markdown doc not on the verified-current list:

- Verify it against the actual code, project files, configuration, build output, or a fresh test run. Do not paraphrase a doc claim into a response without that verification.
- If the statement is wrong or out of date, update the doc file in the same commit as the code change that reveals the discrepancy. Do not leave a known-wrong doc file in place.

This file (`AGENT_CONTEXT.md`) is the one doc that future sessions are *required* to read first, so the bar for keeping it current is correspondingly higher: every commit that invalidates anything in it MUST update it in the same commit, and the maintenance log at the bottom must record the change.

### Terminal commands

- **Single physical line, always.** Chain with `;`. No here-strings, no backticks, no multi-line.
- The user's shell is **PowerShell 7+ (`pwsh.exe`)**. Other shells will be flagged immediately.
- Length is not a concern — correctness and single-line execution are.
- Occasionally a tooling display artifact mangles the leading characters of the command in the echoed output (e.g. `Set-Location` shown as `t-Location`). This indicates the command was not run through the terminal tool as intended. If you see this, stop and fix the command rather than proceeding with a broken one.

### Git commits

- Commit immediately at logical stopping points.
- Use `git add .; git commit -m "..."` chained on one line, with multiple `-m` flags (one per paragraph). **Never** write the message to a file. **Never** use `git commit -F`.
- **No emoji, no non-ASCII** in commit messages.
- Escape `"` and `\` properly so the command parses cleanly.
- **Never run `git push` under any circumstances.** Pushing is a manual step the user performs.
- **One logical change, one commit.** Do not split a single change across multiple commits to work around tool limitations (e.g. the placeholder-hash trailer pattern previously used for `35536b0`, `4364faf`, etc. is retired -- see hash policy below).

#### Commit message hygiene

Commit messages exist to explain the substance of the change to a reader who is browsing history months later. They are NOT a session-narration log, a process audit, or a checklist re-statement. Write them accordingly:

- **Describe what changed and why.** Code changes, the bug being fixed, the design decision being made, the new feature being introduced, performance numbers if relevant, the test count delta if it moved.
- **Do NOT mention things that did not change.** "docs/SETUP.md was checked, no update needed" is noise -- the absence of a docs/SETUP.md edit in the commit already conveys that. Same for any "I checked X and X was fine" line.
- **Do NOT restate rules that live in this file.** "Commit is local; per the never-push rule it will not be pushed" is implicit. "Pre-commit checklist items 1, 2, 4 did not fire" is process narration. "Per the verified-current-docs maintenance protocol" is a process reference. The reader already knows the rules; the commit message describes the change, not the process that produced it.
- **Do NOT refer to yourself as the agent.** Write in the impersonal third person about the change. "Added the guard" not "the agent added the guard". "Restructured the conditional allocation" not "I restructured". The author of the commit is recorded in the git metadata; it does not belong in the message body.
- **The same hygiene applies to AGENT_CONTEXT maintenance-log entries** -- they describe what changed in this file and the load-bearing reason, not the dialogue that led to the change. "User asked the agent to look at X" is the wrong shape; "X was changed because Y" is the right shape.

#### Commit hashes in `AGENT_CONTEXT.md`

The recent-commit-history table and Completed-workstreams entries reference commit hashes as a navigation aid. Because a commit cannot embed its own hash, follow this rule for new B-numbered commits:

- Add the new row to the table with a `<pending>` placeholder where the hash would go. Do the same for the matching Completed-workstreams entry.
- The next session that touches `AGENT_CONTEXT.md` for any other reason replaces any `<pending>` placeholders it finds with the actual hashes from `git log --oneline | Select-String 'B<n>:'`. This is a low-priority janitorial step, not a separate commit.
- **Never** commit a placeholder hash and then immediately follow up with a trailer commit whose only purpose is to fill in the hash. That pattern (used by `35536b0`, `4364faf`, `75ceaa9`, `d5fbc38`, `d3e60d4`) inflated the commit history with no substance and is retired.

### Tool usage

- Prefer `replace_string_in_file` / `multi_replace_string_in_file` over re-creating files. Re-create only when the file is small or being rewritten end-to-end.
- Read files before editing them.
- The `nuget_get-nuget-solver*` and `start_modernization` tools are NEVER appropriate for this project's typical work — do not invoke them unless the user explicitly asks for a package upgrade or .NET version change.
- Do not call `code_search` in parallel; do not call `run_command_in_terminal` in parallel.
- Long-running operations (benchmarks, large test sweeps) may exceed the terminal tool's timeout (whatever it is in the current environment). BenchmarkDotNet writes its result artifacts on each child-process completion, so even when the orchestrating call appears to time out you can usually find the artifacts under `BenchmarkDotNet.Artifacts/results/`. Always check there before assuming a benchmark didn't run.

## Architectural facts to load into working memory

These are non-obvious and have caused real bugs when forgotten.

1. **`SqliteBackend` is single-connection, single-threaded.** Post-B23 every read AND write goes through one `SQLiteConnection` serialized by a single `_writeLock`. Adding more file-level parallelism cannot help SqliteBackend throughput; it just adds contention. This is a known constraint of the chosen backend, not a bug.

2. **`FileOperationDiagnostics` (`.diag` files)** are written by 4 producers: `BackupOrchestrator.BackupFileAsync`, `RestoreService.RestoreFileAsync`, `RestoreService.Batch.cs` (CorruptedRecovery), `IntegrityCheckService.CheckOneFileAsync`. Post-B23 + B24, **all four only emit on real errors**. The `FlushAllLive` shutdown hook in `FileOperationDiagnostics` snapshots any diag still in the live `_live` registry at process exit — so the success/cancel paths in producers MUST call `Discard()` to deregister.

3. **`MemoryBudget` is a soft cap, configured via the WPF app's memory-limit slider** (`MemoryLimitEnabled` + `MemoryLimitMB` in the Configuration table). When `MemoryLimitEnabled=false` the budget is unlimited and the orchestrator can hold many GB of in-flight chunk buffers — confirmed by production telemetry showing multi-GB process working sets on large workloads with `memoryBudget=unlimited`. The dominant memory cost is `EffectiveMaxParallelChunkUploads (default 6) × per-chunk size (up to 64 MB for media) × EffectiveMaxParallelFileBackups (default 8)`.

4. **CPU is NOT the production bottleneck.** Production telemetry on a 7,313-file backup showed CPU usage well below 5%. The actual bottleneck is somewhere downstream: disk read throughput, the SqliteBackend write lock, or HttpClient/TLS state. Any "make this more parallel" hypothesis must be measured rather than reasoned about — the benchmarks under `benchmarks/AzureBackup.Benchmarks/` exist precisely for this.

5. **`InMemoryBlobService`** (`src/AzureBackup.Core/Services/InMemoryBlobService.cs`) is a production-quality fake of `IBlobStorageService`. Used by ~10 test files and by all 4 backup benchmarks. Constructor takes optional `simulatedLatencyMs` and `failureRate`. Honors deduplication. Tracks `TotalBytesUploaded` / `TotalOperations`. **This is the single biggest enabler of network-free testing in the project.** Use it instead of inventing a new mock.

6. **Backend selection is SQLite-only in production.** As of B58 the legacy `AZBK_USE_SQLITE` env-var feature flag and every code path it gated have been removed. `DatabaseBackendFactory.ShouldUseSqlite()` now returns `true` unconditionally except when an internal-only `AsyncLocal<bool?>` test override pins it to `false`; production callers cannot reach the LiteDB branch and there is no env var to set or unset. The override exists only so the `BackendOverrideScope` test helper and the `BackendContractTests` shared-contract suite can still exercise the retained `LiteDbBackend` adapter. Do NOT re-introduce an environment variable to flip backends; do NOT add new production callers of the AsyncLocal override; do NOT delete `LiteDbBackend` until the contract-test suite is also retired.

7. **`BackupOrchestrator` constructor** signature: `(LocalDatabaseService, EncryptionService, ChunkingService, IBlobStorageService, FileWatcherService)`. The `FileWatcherService` is required even when no folders are watched — it can be constructed without ever being started.

8. **`BackupOrchestrator` parallelism overrides (B25-bench-2 seam):** two optional `int?` properties — `MaxParallelChunkUploadsOverride` (default null → 6) and `MaxParallelFileBackupsOverride` (default null → 8). Read once per backup operation via internal `EffectiveMax*` getters so they cannot change mid-pipeline. Production behavior is unchanged when overrides are unset; the seam exists for the parallelism benchmarks. If you want to change the production *default*, change the constants in `BackupOrchestrator.cs`, not the override.

9. **`RestoreService.SmallFileThresholdBytes` (B26) is the single source of truth for what "small" means.** This `public const long` (currently `16L * 1024 * 1024`, i.e. 16 MB) lives on the `RestoreService` partial in `src/AzureBackup.Core/Services/RestoreService.Batch.cs` and is referenced by **both** the production restore pipeline (which routes files at or below the threshold through a 32-way parallel small-file lane) **and** the desktop UI (which groups the same set of files into the aggregate "Small files" progress row in `ProgressTabViewModel.SmallFileGroupText`, plus three call sites in `MainWindowViewModel.Commands.Backup.cs` / `MainWindowViewModel.Commands.TreeView.cs`). Before B26 the UI hard-coded 100 MB in three places while production used 16 MB, so the progress row mis-grouped a strictly larger set of files than the pipeline actually optimized for. **Do not introduce a new hard-coded copy of this value anywhere.** If you need the threshold in a new file, add `using AzureBackup.Core.Services;` and reference `RestoreService.SmallFileThresholdBytes` directly. The displayed user-facing label interpolates the constant so it tracks any future change automatically.

10. **B27 benchmark seam (`BackupBenchmarkBase`) — three new public surfaces.** `BackupBenchmarkBase.MemoryLimitMBOverride` is a `protected virtual int?` returning `null` by default; when non-null, `BaseIterationSetup` writes `MemoryLimitEnabled=true` + `MemoryLimitMB=<value>` via `LocalDatabaseService.SaveConfiguration` BEFORE the orchestrator runs, and `BackupOrchestrator.BackupFilesCoreAsync` picks it up via its `GetConfiguration()` call at the top of every backup. This is how the big-scale benchmarks simulate the recommended 16 GB production setting without touching production code. `BackupBenchmarkBase.BigWorkloads` is the public set of disk-heavy workloads (`media-library-500` ~260 GB, `production-scale-3000` ~70 GB, `huge-outlier-mixed` ~32 GB) gated by a new disk-space preflight at the top of `BaseGlobalSetup` that throws a clear error if free disk < 1.5x estimated workload size. The preflight reads `EstimateWorkloadBytes(Workload)` which returns 0 for the pre-B27 small workloads (no preflight) and a conservative overestimate for big ones. **Pre-B27 small workloads are unchanged** so the historical i7-9700K result tables in the four pre-B27 benchmark xmldocs remain directly comparable to a fresh small-workload run on the same hardware. Do not duplicate the `MemoryLimitMB=16384` value — if the recommended production setting changes, change it once in the relevant `*BigScaleBenchmark.cs` files (and `ProductionScaleBackupBenchmark.cs` and `MemoryBudgetBenchmark.cs`) and update this fact. **2026-04-27 correction (post-B27 hardware run):** the seam works as designed; the OOM observed at `[ChunkConcurrency=24, Workload=media-library-500]` on the first B27 hardware run was NOT caused by `MemoryLimitMB=16384` failing to bind. Root cause was the destination side -- see fact #11.

11. **`InMemoryBlobService` discard mode (B27).** New constructor parameter `bool retainPayloads = true` (default preserves all pre-B27 behaviour). When `false`, `UploadChunkAsync` / `UploadChunkDirectAsync` still record the blob name and tier in `_blobs` / `_blobTiers` (so dedup, listing, and `BlobExistsAsync` work) but the encrypted payload is replaced with `Array.Empty<byte>()`; the would-be storage cost is recorded in `_discardModeStorageBytes` so `TotalStorageUsed` stays accurate. Discard mode short-circuits the dedup-collision verification path because the original bytes are no longer present to compare against -- it trusts the hash, matching real Azure's optimistic dedup. `DownloadChunkAsync` and `VerifyChunkIntegrityAsync` are not supported against a discard-mode store, so restore-style benchmarks must NOT opt in. Production code never sets the flag; production tests never set the flag. The flag exists exclusively to remove the RAM ceiling on big-scale backup-throughput benchmarks: without it, `media-library-500` (~260 GB) cannot fit its encrypted ciphertext in process memory regardless of `MemoryBudget`, and the previously observed 58 GB peak working set was the `_blobs` dictionary growing unboundedly, not the orchestrator chunk-buffer pool. The benchmark opt-in is the new `protected virtual bool BackupBenchmarkBase.RetainBlobPayloads => true`; all five big-scale benchmark classes (`AdaptiveChunkConcurrencyBigScaleBenchmark`, `TwoTierFileSplitBigScaleBenchmark`, `LargestFirstSchedulingBigScaleBenchmark`, `ProductionScaleBackupBenchmark`, `MemoryBudgetBenchmark`) override it to `false`. **DO NOT add a new big-scale benchmark against `media-library-500` or larger without setting `RetainBlobPayloads => false`** -- it will OOM the host. **DO NOT use discard mode for any test that calls `DownloadChunkAsync` / `VerifyChunkIntegrityAsync` against the same store** -- they will fail because the bytes are gone.

12. **`BackupOrchestrator.BackupFilesAsync` reports unchanged-file bytes as completed (B31).** The orchestrator computes its own `totalFiles` and `totalBytes` from the input list it receives (`BackupFilesCoreAsync` lines 681-696). When a file is detected as unchanged via the metadata-skip path (`BackupOrchestrator.cs` ~line 648, `progress?.Report((fileInfo.Length, fileInfo.Length))`), the unchanged file's full size flows into the running `bytesProcessed` essentially instantly. This is INTENTIONAL and is the behaviour the file-watcher path and `MirrorSyncToAzureAsync` rely on. Consequence for UI callers that derive their progress denominators from a preview's `TotalBytesToTransfer` (which excludes unchanged files): if the UI passes the unfiltered selection list to `BackupFilesAsync`, the UI's preview-derived denominator (e.g. 1 TB of changed bytes) will be exceeded by the orchestrator's reported numerator (e.g. 2 TB including unchanged), producing 200%-style progress and a wildly wrong upload-speed estimate that briefly shows the unchanged files' bytes as transferred instantaneously. **DO NOT call `BackupOrchestrator.BackupFilesAsync` from a UI command path with the unfiltered user selection.** The B31 fix makes `MainWindowViewModel.ConfirmAndFilterBackupFilesAsync` rebuild the file list as `preview.FilesToCreate.Where(IsIncluded) UNION preview.FilesToOverwrite.Where(IsIncluded)` so unchanged files are stripped before the orchestrator ever sees them. Two regression tests in `BackupOrchestratorTests` (`BackupFilesAsync_UnchangedFileInInputList_OrchestratorReportsItsBytesAsCompleted` and `BackupFilesAsync_MixedChangedAndUnchanged_BytesProcessedSumsAllInputs`) pin the orchestrator-side behaviour the UI fix relies on. The Mirror Sync path on `MainWindowViewModel.Commands.TreeView.cs` is structured differently (it derives its denominator from the same unfiltered list and adds unchanged files instantly via an explicit "Unchanged" handler) and is not affected by this fix.

13. **Force-reupload + integrity auto-repair (B42).** `BackupOrchestrator.BackupFileAsync` gained a new optional parameter `bool forceReupload = false` on its deepest overload (the three shorter overloads default to `false` and behave exactly as before). When `true`, the metadata-skip fast path is bypassed (~line 654), `existingHashes` is treated as empty so the chunker emits every chunk as new (~line 681), and `isNewFile` is forced so the upload routes through `UploadChunkDirectAsync` instead of the dedup-aware `UploadChunkAsync` path (~line 686). `BackupOrchestrator.BackupFilesAsync` and `BackupFilesCoreAsync` both accept and propagate the same flag. **`PerformFullScanAsync` (in `BackupOrchestrator.Operations.cs`) was rewired in the same commit to actually re-upload** -- pre-B42 it only enqueued `FileChangeEvent`s which still hit the metadata-skip path for unchanged files, despite the UI button being labelled as a force-reupload action. It now invokes `BackupFilesAsync(allFiles, ..., forceReupload: true, ct)` directly. `IntegrityCheckService` gained a public `Func<string, CancellationToken, Task<bool>>? RepairCallback` and a private `RepairableReasons` whitelist (`missing-blob`, `wrong-size`, `md5-mismatch`, `crc-mismatch`, `decrypt-failed`, `byte-differ`, `download-error`); `IntegrityCheckOptions` gained `bool AutoRepairOnFailure { get; init; } = true`; `IntegrityCheckRun` gained `int FilesAutoRepaired` (B43: now persisted to the SQL `integrity_check_runs.files_auto_repaired` column; pre-B43 rows backfill to 0). When the first per-file pass produces at least one repairable failure and a callback is wired, the engine `Discard()`s the first-pass diag, invokes the callback (which is wired in `MainWindowViewModel` to `_orchestrator.BackupFileAsync(..., forceReupload: true, ...)`), re-reads the file's metadata via `_databaseService.GetBackedUpFile(path)` so the second pass uses the freshly-uploaded chunk hashes, and re-runs the chunk-loop. If the second pass is clean: no `.diag` is written, no row is persisted in `integrity_check_failures`, and `FilesAutoRepaired` is incremented. If the callback returns `false` or the second pass still fails: the original (or post-repair) failures are reported normally with a `.diag` flushed via the new `ReplayFailuresToDiag` / `FlushPassDiag` helpers. The retry is one-shot -- no recursion, no escalating budget. Three regression tests in `IntegrityCheckServiceTests` pin the contract (`AutoRepair_MissingBlob_ReuploadFixesIt_NoFailureRecordedNoDiag`, `AutoRepair_Disabled_FailureStillReported`, `AutoRepair_CallbackReturnsFalse_OriginalFailureReported`). **DO NOT call `BackupFileAsync(path, ct)` (the two-arg overload) from a repair callback** -- it hits the metadata-skip path and the missing/corrupt blob stays missing/corrupt; you MUST use the deepest overload with `forceReupload: true`. **DO NOT add a new failure reason that re-uploading cannot fix to `RepairableReasons`** (e.g. `local-file-missing`, `engine-error`) -- the callback would loop pointlessly and the user would see no failure.

17. **In-place narrow REINDEX-based catalog repair (B45).** `SqliteBackend.IntegrityFile.cs` adds `ReindexCorruptIndexes(DatabaseFileIntegrityResult diagnosis)` plus the supporting `DatabaseRepairClassifier`, `DatabaseRepairClassification`, and `DatabaseRepairResult` types. 24. **Budget-derived `LargeChunkBufferPool` global cache cap (B52).** The B37 recycler exposed a hidden second residency ceiling: its per-bucket cap of 32 multiplied across all five bucket sizes allows up to ~15.5 GB of LOH-resident `byte[]` to sit in the pool's `ConcurrentBag` arrays, completely independent of the active `MemoryBudget`. A user with `MemoryLimitMB=8192` could therefore see process working set climb well past the configured budget on a media-heavy workload, because the budget only throttles in-flight admission while the recycler retained idle buffers indefinitely. B52 introduces a second constructor `LargeChunkBufferPool(long maxCachedBytes)` (the existing parameterless constructor forwards to `long.MaxValue` to preserve all pre-B52 callers including tests and benchmarks). When `Return(byte[])` would push `TotalBytesCached + bucketSize` above `MaxCachedBytes`, the buffer is dropped on the floor instead of being cached and `TotalReturnsDroppedForCap` is incremented; the per-bucket cap and the zero-out-on-return defence are unchanged. Production callers in `BackupOrchestrator.BackupFilesAsync` and `BackupOrchestrator.MirrorSyncToAzureAsync` derive the cap via the new private static `BackupOrchestrator.ComputePoolCapBytes(MemoryBudget budget)` helper, which returns 25 percent of the configured budget with a 256 MB floor (one largest-bucket-size, so the cap can never starve the pool of even one entry) and `long.MaxValue` for unlimited budgets. The 25 percent ratio was chosen because the recycler exists to recycle buffers across chunks; the in-flight work is what `MemoryBudget` itself throttles, so allocating the recycler more than a quarter of the budget would force admission to stall even when no real work was outstanding. Seven new regression tests in `LargeChunkBufferPoolTests` (`DefaultConstructor_HasUnlimitedGlobalCap`, `GlobalCap_DropsReturnsThatWouldExceedCap`, `GlobalCap_AcceptsAgainAfterRentDrainsResidency`, `GlobalCap_PerBucketCapStillApplies`, `GlobalCapZero_Throws`, `GlobalCapNegative_Throws`, `Dispose_ResetsResidencyButRetainsCap`) pin the new contract; the `GlobalCap_PerBucketCapStillApplies` test specifically asserts that per-bucket overflows are NOT counted under `TotalReturnsDroppedForCap` because the two caps measure different rejection reasons. **DO NOT remove the parameterless `LargeChunkBufferPool()` constructor** -- the existing benchmark seam in `BackupBenchmarkBase` and several test fixtures rely on the pre-B52 unbounded shape; the constructor exists so the global cap is opt-in for callers that explicitly want the old behaviour. **DO NOT compute `ComputePoolCapBytes` from `_databaseService.GetConfiguration()` directly** -- the `MemoryBudget` instance is the single source of truth for the active budget within an operation scope, and reading the config separately would let a slider change mid-operation produce a pool cap that disagrees with the budget the same operation is throttling against. **DO NOT lower the 256 MB floor** -- the largest bucket is 256 MB, so a smaller floor would mean a cap that cannot accept even one largest-bucket entry, effectively disabling the pool on small budgets where the recycler benefit is most visible because the GC pressure is highest. **DO NOT raise the 25 percent ratio without a fresh `MemoryBudgetBenchmark` run** -- the ratio trades cache residency against admission throughput, and the existing big-scale benchmark numbers were taken with the unbounded pool; a higher ratio risks regressing throughput by inflating stalls.

25. **Chunk-size-gated Azure upload transfer options (B53).** The pre-B53 `AzureBlobService.UploadEncryptedChunkAsync` applied a single `DefaultTransferOptions` (`MaximumConcurrency=8`, `MaximumTransferSize=8 MB`) to every encrypted chunk regardless of size, so the SDK staged ~64 MB per in-flight upload (`MaximumConcurrency * MaximumTransferSize`) even for chunks small enough to fit in a single PUT. The 8 MB block size and 8-way ceiling are preserved verbatim for the upper band so the per-chunk throughput on chunks that actually need parallel staging is unchanged; only chunks small enough to fit in one or two blocks see reduced staging. 26. **Budget-derived effective file concurrency (B54).** Pre-B54 `BackupOrchestrator` used `EffectiveMaxParallelFileBackups` (default 16) as the file-level fan-out for every backup operation, regardless of the active `MemoryBudget`. On a small `MemoryLimitMB` setting that constant could admit more files than the budget can sustain, causing every chunk producer to stall against the budget the moment it tried to acquire a chunk buffer; the orchestrator would still SPIN UP all 16 file workers, each contributing Azure SDK staging residency outside the budget. B54 derives the effective file concurrency from the active budget via the new `internal static int BackupOrchestrator.ComputeEffectiveFileConcurrency(MemoryBudget budget, int configuredCeiling)` helper, which returns `clamp(budget.TotalBytes / EstimatedPerFileResidencyBytes, 1, configuredCeiling)` for finite budgets and `configuredCeiling` for unlimited budgets. The estimate `EstimatedPerFileResidencyBytes = 512 MB` matches the worst-case per-file producer charge (`MaxParallelChunkUploads (6) × MaxChunkSize (64 MB) = 384 MB` plus per-operation overhead) and also matches the historical B27 hand-picked ratio (`MemoryLimitMB=8192 ÷ MaxParallelFileBackups=16 = 512 MB per file`); the formal scaling rule therefore agrees with the historical default at the production setting and only changes behaviour at smaller budgets. The floor of 1 preserves the invariant that backups always make some forward progress even on a tiny budget; the budget itself still throttles admission inside the per-file pipeline. Three call sites are wired: `BackupFilesAsync` snapshots the value once and passes it through the new `BackupFilesCoreAsync` parameter `int effectiveFileConcurrency`; `MirrorSyncToAzureAsync` does the same so the mirror path's log line, metrics (`OperationMetrics.FileConcurrency`), and `ParallelOptions.MaxDegreeOfParallelism` all agree on the same value (the variable was hoisted out of the `if (filesToBackup.Count > 0)` block so the post-loop metrics record the value the loop actually used); `RunBackupLoopAsync` computes a fresh `batchFileConcurrency` per loop iteration so a slider change between batches takes effect immediately. The `Metrics?.RecordDecision("backup-concurrency", ...)` payload now carries both `maxParallelFileBackups` (the hard ceiling) and `effectiveFileConcurrency` (the budget-clamped value) so a post-hoc analysis can attribute throughput deltas to either the constant or the budget. Fourteen new regression tests in `tests/AzureBackup.Tests/Unit/BackupOrchestratorFileConcurrencyTests.cs` (`UnlimitedBudget_ReturnsConfiguredCeiling`, `BudgetWellAboveCeiling_ReturnsConfiguredCeiling`, `BudgetEqualsCeilingTimesPerFile_ReturnsCeilingExactly`, parameterised `BudgetBelowOrAtCeiling_ScalesDown` across 8 GB / 4 GB / 2 GB / 1 GB / 512 MB / 256 MB tiers, `TinyBudget_ReturnsFloorOfOne`, `NullBudget_Throws`, `ZeroCeiling_Throws`, `NegativeCeiling_Throws`, `CeilingOfOne_AlwaysReturnsOne`) pin the contract. **DO NOT compute `ComputeEffectiveFileConcurrency` from `_databaseService.GetConfiguration()` directly** -- the `MemoryBudget` instance is the single source of truth within an operation scope, and reading the config separately would let a slider change mid-operation produce a fan-out that disagrees with the budget the same operation is throttling against. **DO NOT lower `EstimatedPerFileResidencyBytes` below 512 MB** -- the value matches both the worst-case per-file producer charge and the production-tested B27 ratio, and a lower estimate would over-admit files at the production budget and re-introduce the W3 over-budget condition. **DO NOT remove the floor of 1** -- a tiny-budget user must still be able to back up; the budget itself will throttle admission inside the per-file pipeline. **DO NOT change the helper to read `Environment.ProcessorCount`** -- file-level fan-out is a memory-residency decision, not a CPU decision, and CPU is not the production bottleneck (architectural fact #4); blending CPU into this calculation would re-introduce the pre-B27 throttling pattern that the hardware run already disproved. **DO NOT bypass the helper by passing `EffectiveMaxParallelFileBackups` directly to `ParallelOptions.MaxDegreeOfParallelism`** in any new operation entry point -- the helper is the single point that enforces the budget-vs-fan-out relationship, and a direct reference re-introduces the pre-B54 admission gap.

27. **Producer-side Azure SDK staging charge (B55).** The B53 chunk-size-gated upload helper made the SDK staging residency *predictable* per chunk, but pre-B55 the staging bytes were still entirely invisible to the active `MemoryBudget`: `ChunkingService.AcquireChunkBufferAsync` charged only the chunk payload (B30) plus the encrypt-side ArrayPool rental (B38), so a 16-way file × 6-way chunk pipeline could push process working set well above `MemoryLimitMB` even when the budget read near-empty. B55 closes the gap by adding the new `internal static long AzureBlobService.EstimateUploadStagingBytes(int encryptedLength)` helper and folding its return value into `AcquireChunkBufferAsync`'s atomic `MemoryBudget.AcquireAsync`. The helper is implemented as `MaximumConcurrency × MaximumTransferSize` from the same `ComputeUploadTransferOptions(encryptedLength)` that the upload site itself hands to the Azure SDK -- the two numbers move in lockstep by construction, so a future change to the per-band staging shape automatically updates both the upload site's residency and the budget's accounting. Per-chunk staging charges therefore range from 1 MB (≤ 8 MB chunks, single PUT) up to 64 MB (> 32 MB chunks, full 8×8 MB fan-out). The producer-side charge is a conservative upper bound (the SDK does not necessarily hold the full N×M block at every instant) but matches the same upper-bound discipline B30/B38 already use for the chunk and encrypt buffers. The deadlock-avoidance branch in `MemoryBudget.AcquireAsync` (B34) still admits an at-least-one oversized request, so a tiny budget cannot wedge the pipeline. The per-file residency estimate `EstimatedPerFileResidencyBytes = 512 MB` (architectural fact #26 / B54) was deliberately NOT widened to match the new larger per-chunk charge -- raising it would over-throttle file-level fan-out and re-introduce the pre-B27 admission stalls; instead, the budget itself admits fewer concurrent chunks per file when staging is charged, which converges to the same in-flight residency without an orchestrator-side over-correction. Nine new tests in `AzureBlobServiceTransferOptionsTests` parameterised across 1 MB / 8 MB / 16 MB / 32 MB / 64 MB / 128 MB / 256 MB pin `EstimateUploadStagingBytes` to the same residency values the staging-bound test pins for `ComputeUploadTransferOptions`, plus zero/negative input validation; this guarantees the budget charge cannot drift away from the actual SDK staging shape used at the upload site. **DO NOT compute the staging charge from a separate constant** -- the helper deliberately re-uses `ComputeUploadTransferOptions` so that any future change to the per-band staging shape stays in sync between the upload site and the budget; a duplicated constant would silently over- or under-charge on the next B53 ladder change. **DO NOT widen `EstimatedPerFileResidencyBytes` to compensate for the larger per-chunk charge** -- the budget itself absorbs the extra charge by admitting fewer chunks per file; raising the per-file estimate would over-throttle file fan-out and undo the historical B27 production-default sizing. **DO NOT push the staging charge to the consumer side** -- B38 already proved that any consumer-side acquire on the same budget can deadlock against a producer that holds its charge until the consumer finishes; the producer-side charge is the only deadlock-free architecture for pipeline residency. **DO NOT skip the staging charge for chunks below the pool-skip threshold** -- the SDK stages the same per-band shape regardless of whether the chunk buffer came from the ArrayPool or a fresh allocation, so the staging charge is independent of the pool-routing decision and must apply to both code paths.

29. **`QuarantineFileLocator` codifies pair-by-stamp invariants for the rebuild form (B61).** New `public static class QuarantineFileLocator` in `src/AzureBackup.Core/Services/QuarantineFileLocator.cs` is the single source of truth for "which quarantined file pair should the Settings rebuild form auto-fill?". 31. **B63 polish on the B62 restore pipeline (variable-step AIMD, peak telemetry, watchdog deduplication, cleanup correctness).** Follow-up to architectural fact #30 after a code review of the just-landed B62 pipeline. Five coordinated changes, all in the restore code path. (a) **Variable-step additive increase.** `BandwidthScheduler.ApplyAdditiveIncrease(double delta)` now steps `+2` when `delta > PlateauBand * 4` (a strong-improvement signal) and `+1` otherwise, instead of always `+1`. AIMD's "additive" half is conventionally +1 per evaluation tick, but on a fast link a +1 climb from `initialConnections=4` to a 16-32 ceiling takes 24-56 s of sustained sample windows; on a household 100 Mbps link the user-perceived ramp under the pre-B63 +1 step was visibly slow at the start of a large restore. The +2 step is bounded by the existing `_maxConnections` clamp and only fires when the EWMA improvement is at least 4x the plateau band so a noisy single sample cannot trigger overshoot. The increase remains "additive" in character (linear, not multiplicative) -- the change is the linear coefficient, not the policy. (b) **`PeakActiveSlots` lifetime high-water mark.** New `BandwidthScheduler.PeakActiveSlots` `int` property tracks the maximum `_activeSlots` value observed over the scheduler's lifetime, updated lock-free via a CAS loop inside `AcquireSlotAsync`. Surfaced so `RestoreFilesBatchCoreAsync` can record the actual peak file-level concurrency in `OperationMetrics.FileConcurrency` rather than the static `MaxParallelFileRestores` ceiling, which post-B62 no longer reflects what the AIMD controller actually used. New `RestoreResult.PeakFileConcurrency` `int` carries the peak through `RestoreFilesWithRemappingAsync` and `MirrorSyncToLocalAsync`; both metric-emit sites use the peak when non-zero (scheduler engaged at least once for a large file) and fall back to `MaxParallelFileRestores` when zero (small-file-only batch never engaged the scheduler). (c) **Wakeup semaphore deduplication.** `BandwidthScheduler.ReleaseSlot()` now suppresses the wakeup when the post-decrement active count is still at or above the allowed ceiling. Pre-B63 every `ReleaseSlot` woke one waiter, which then re-parked because `active >= allowed`; under multiplicative-decrease pressure with many parked waiters this produced quadratic wakeup churn. Eventual additive-increase steps still wake waiters via their own `_acquireWakeups.Release(N)`, so skipping the redundant wakeup here is safe. The internal field was renamed from `_slotReleased` to `_acquireWakeups` because the semaphore is now released both when a slot is given back AND when an additive-increase step grants new capacity, and the old name implied only the first cause. The semaphore release for additive-increase wakeups is now also computed under `_evaluationLock` and released AFTER the lock exit -- pre-B63 the release fired under the lock, which was fragile against any future continuation that might re-enter scheduler code. (d) **Watchdog snapshot under the writer's lock.** `RestoreWithBoundedParallelDownloadsAsync` now hoists the writer's `pendingChunks` `Dictionary<int, (byte[] data, int length)>` to the outer scope, paired with a new `pendingChunksLock` `Lock`, and the stall watchdog snapshots it directly under that lock. Pre-B63 the writer maintained a parallel `ConcurrentDictionary<int, byte> bufferedIndicesSnapshot` purely so the watchdog could read indices without taking the writer's lock; that doubled per-chunk bookkeeping cost on the hot path and could drift from `pendingChunks` if the writer ever forgot to mirror an add/remove. The watchdog poll cadence is `WatchdogPollMs = 5_000`, so briefly blocking the writer for the snapshot has no measurable throughput impact and removes a whole class of "the parallel dictionary drifted" bugs. (e) **Producer cleanup correctness on cancel/error.** The producer's `catch (OperationCanceledException)` and `catch (Exception ex)` branches in the per-chunk download `Task.Run` lambda now release ONLY the budget bytes still owed (`owedBudget`, decremented to zero after the channel write transfers ownership to the writer) and return the rented buffer ONLY when it is still held (`rentedBuffer != null`, niled out after the channel write succeeds). Pre-B63 the catches blindly released `chunkMemoryCost` and never returned the rented buffer, which both over-released budget when the exception fired between Phase B (post-`memoryBudget.Release(chunk.Length)`) and the channel write, and leaked ArrayPool buffers on every cancelled or failed download that had already produced a chunk. The watchdog also stamps `lastDownloadCompletedTicks` BEFORE the bounded-channel `WriteAsync` so the "downloads idle" line distinguishes a download-stage stall from a writer-stage stall (a slow writer parking the producer in `WriteAsync` is a writer stall, not a download stall, and conflating the two would mis-classify the watchdog snapshot). The writer's `WriteChunkAndReleaseAsync` now wraps the body in a try/finally with a `windowSlotReleased` flag so a transient `FileStream.WriteAsync` exception cannot leak a window slot and stall the next chunk; the inline release on the success path stays for clarity, the finally is belt-and-braces. New tests in `BandwidthSchedulerTests` cover the new step policy and `PeakActiveSlots`. Tests: 907 of 907 pass. Build: 0 warnings, 0 errors. `docs/USER_GUIDE.md` reviewed per the verified-current-docs maintenance protocol -- restore scheduling is internal and the user-visible memory-limit contract is unchanged. `docs/SETUP.md` reviewed -- no new build flag, runtime identifier, package, schema column, encryption-envelope change, or `AppMode.DataDirectory` resolution change. **DO NOT raise the `+2` step above `+2` or remove the `delta > PlateauBand * 4` gate** -- a larger step or a wider gate erodes the "additive" half of AIMD and can overshoot the bandwidth-fitting ceiling, defeating the whole controller. **DO NOT reset `PeakActiveSlots` between batches** -- the value is the lifetime high-water mark and is consumed at scheduler-instance disposal; resetting it would make `OperationMetrics.FileConcurrency` under-report on long-running batches. **DO NOT remove the wakeup-suppression branch in `ReleaseSlot()`** -- it is the structural defence against the quadratic wakeup churn under multiplicative-decrease pressure; the additive-increase wakeup path is independent and still fires its own `_acquireWakeups.Release(N)`. **DO NOT release `_acquireWakeups` from inside `_evaluationLock`** -- a future continuation that re-enters scheduler code could deadlock; compute the wakeup count under the lock and release after exit. **DO NOT reintroduce a parallel `ConcurrentDictionary` mirror of `pendingChunks` for watchdog purposes** -- the locked snapshot under `pendingChunksLock` is the canonical read path, and any mirror is an opportunity for drift. **DO NOT blindly release `chunkMemoryCost` in producer-side catches** -- track exactly how much budget is still owed via the `owedBudget` local and release that, otherwise the budget over-releases when the exception fires after Phase B; pair the same discipline with the `rentedBuffer` local so the ArrayPool buffer is returned exactly once across all paths.

## Conventions / patterns observed in this codebase

- **`[Conditional("DIAGNOSTICLOG")]` logging**: services use `Log(string)` methods that are completely no-ops in Release without the `DIAGNOSTICLOG` constant set. The B16 commit also adds a title-bar marker showing whether the build has DEBUG/DIAG on.
- **Diagnostic events forwarded by name**: services raise `EventHandler<string>? DiagnosticLog` events; `MainWindowViewModel.OnDiagnosticLog` aggregates them all.
- **Non-destructive catalog quarantine for crash recovery**: an unreadable on-disk catalog is moved aside under a timestamped `.quarantine-yyyyMMdd-HHmmss` suffix (the bytes are NEVER securely deleted) so a fresh catalog can be initialized at the same path. See `LocalDatabaseService.Migration.cs` (`QuarantineCorruptDatabase`, `QuarantineAndClose`) and `BackupOrchestrator.RebuildFromQuarantinedCatalogAsync`. Pre-B58 the same partial also held a `.upgrade-pending` sentinel + four-step rename dance for the LiteDB-to-SQLite migration; that surface was retired with the W4 cleanup and is no longer a reference for new file-rename code.
- **`PaddedLong` for hot atomic counters**: cache-line-padded structs prevent false sharing. See `BackupOrchestrator` line ~592 for example. If adding new hot counters under parallel workers, use `PaddedLong`.
- **`ArrayPool<byte>` discipline**: producer/consumer code transfers ownership of pooled buffers via the channel. The `ChunkPayload` record carries the rented buffer and the consumer is responsible for `ArrayPool<byte>.Shared.Return(payload.Data)` after upload completes. Do not double-return.
- **No emoji** anywhere in the commit messages.
- **xmldoc style is heavy and discursive** — multi-paragraph `<remarks>` blocks explaining WHY a design decision was made, with B-numbers cross-referencing commits. Match this style when adding new public surface.
